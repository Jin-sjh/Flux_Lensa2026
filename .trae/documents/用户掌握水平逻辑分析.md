### ⚠️ 问题 2: 后端 API 缺少认证保护（安全问题）

**问题描述**：`PATCH /api/user/level` 端点没有认证中间件，任何人都可以修改任意用户的水平。

**当前代码** ([auth.py:98-116](file:///d:\迅雷云盘\Flux_Lensa2026\inputable\routers\auth.py#L98-L116)):
```python
@router.patch("/api/user/level", response_model=UserResponse)
async def update_user_level(
    request: UpdateLevelRequest,
    db: AsyncSession = Depends(_get_db()),  # ❌ 缺少认证依赖
):
```

**修复建议**：
```python
from services.auth_service import decode_token
from fastapi.security import HTTPBearer

security = HTTPBearer()

async def get_current_user(token: str = Depends(security), db: AsyncSession = Depends(get_db)):
    payload = decode_token(token.credentials)
    if not payload:
        raise HTTPException(status_code=401, detail="无效的认证令牌")
    user = await db.execute(select(User).where(User.user_id == payload["sub"]))
    return user.scalar_one_or_none()

@router.patch("/api/user/level", response_model=UserResponse)
async def update_user_level(
    request: UpdateLevelRequest,
    current_user: User = Depends(get_current_user),  # ✅ 添加认证
    db: AsyncSession = Depends(_get_db()),
):
    if current_user.user_id != request.userId:  # ✅ 验证用户身份
        raise HTTPException(status_code=403, detail="无权修改其他用户信息")
```

### ⚠️ 问题 3: 两套不一致的水平计算逻辑

**问题描述**：前端和后端使用不同的水平计算方式，可能导致结果不一致。

| 计算方式 | 前端 (testStore.ts) | 后端 (user_model.py) |
|---------|---------------------|---------------------|
| **依据** | 测试分数百分比 | 已知词汇列表 |
| **B1 条件** | score >= 80 | A1>=3 且 A2>=2 且 B1>=1 |
| **A2 条件** | score >= 50 | A1>=3 且 A2>=2 |
| **A1 条件** | score < 50 | 其他情况 |

**潜在问题**：
- 用户在前端完成测试得到 A2，但后端的备用计算可能给出不同结果
- 两套逻辑维护成本高，容易产生不一致

**修复建议**：
1. **统一使用前端计算**：后端只负责存储，不做二次计算（当前实际行为）
2. **或者统一到后端**：前端只提交答案，后端计算并返回结果
3. **文档化**：明确说明两套逻辑的用途和场景

---

### ⚠️ 问题 4: 测试题目与后端词汇不同步

**问题描述**：前端测试题目是硬编码的静态数据，与后端的 `PLACEMENT_WORDS` 没有关联。

**当前状态**：
- 前端测试题：15道题目（词汇+语法）
- 后端词汇：A1(4词) + A2(4词) + B1(2词) = 10词

**潜在问题**：
- 前端测试题目的词汇可能与后端 `PLACEMENT_WORDS` 不一致
- 更新词汇时需要同时修改两处

**修复建议**：
1. 后端提供测试题目 API，前端从后端获取
2. 或者将 `PLACEMENT_WORDS` 与测试题目关联
